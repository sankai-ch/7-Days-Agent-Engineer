## 5. 已知风险与局限

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Token 估算基于字符数，非精确值 | 成本统计有 ±20% 误差 | 生产环境接入 API 返回的 usage 字段 |
| Trace 存储在内存中 | 重启后丢失 | 生产环境替换为 Jaeger/Zipkin |
| 幂等缓存在内存中 | 单机有效，集群失效 | 生产环境替换为 Redis |
| 熔断器阈值固定 | 无法自适应 | 可集成 Resilience4j 自适应熔断 |
| LLM 输出格式不稳定 | 工具调用解析偶尔失败 | 已有 JSON 提取容错 + 重试机制 |
| 示例数据量小 | 评测结果代表性有限 | 接入真实数据后重新评测 |

## 7. 面试叙事参考

### 一句话介绍

> 用 7 天从零构建了一个具备 **RAG 知识库 + MCP 工具调用 + 多角色编排 + 全链路可观测 + 安全防护** 的生产雏形 Agent 系统。

### 技术亮点（面试可展开讲的点）

1. **MCP 协议手动实现**
   - 没有依赖第三方 MCP SDK，手动实现了 JSON-RPC 2.0 协议
   - 注册表模式支持工具热插拔（新工具只需 implements McpTool + @Component）

2. **三角色编排（Planner/Executor/Reviewer）**
   - 相比单 Agent 模式，复杂任务的信息覆盖度从单一维度提升到多维度交叉分析
   - Reviewer 的补充循环机制保证了信息完整性

3. **安全防护体系**
   - 责任链模式串联 5 层安全检查（限流→熔断→幂等→注入检测→脱敏）
   - 19 条攻防用例覆盖 OWASP 常见攻击向量

4. **全链路可观测**
   - 自研 Tracing 框架（Span 树 + ThreadLocal 上下文传递）
   - AOP 无侵入织入，每次请求自动生成完整的调用链
   - Token/Cost 估算让每次 LLM 调用的成本可量化

5. **离线评测框架**
   - 10 条回归用例覆盖 agent 和 orchestrate 两种模式
   - 一个 curl 命令即可触发全量评测，输出成功率/时延/成本

### 常见面试追问

**Q: 为什么不用 LangChain 的 Agent 框架？**
> 手动实现更容易理解底层原理（ReAct 模式、Prompt 工程、JSON 解析容错），也更容易定制安全和可观测能力。

**Q: 三角色编排相比单 Agent 的实际效果？**
> 对于"分析系统健康状况"这类需要从日志+工单两个维度收集数据的任务，单 Agent 只会调一个工具，编排模式会自动拆解为 2-3 步并交叉分析。

**Q: 安全防护如何落地？**
> SecurityChain 在 Controller 层显式调用，前置检查（限流/熔断/注入检测/脱敏）和后置处理（输出脱敏/熔断状态记录/幂等缓存）完全包裹了业务逻辑。19 条攻防测试全部通过。

**Q: 如何保证 LLM 输出格式的稳定性？**
> 三层容错：1) 提示词严格约束输出格式 2) extractJson() 支持 ```json``` / ``` ``` / 裸 JSON 三种格式 3) 解析失败时回退为直接回答。
